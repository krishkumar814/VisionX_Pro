### 🔷 2. Project Overview (Real Understanding)

* What does your project actually do? It's an intelligent educational platform that generates automated topic-specific quizzes, offers highly targeted AI chatbot tutoring from custom teacher PDFs, maps out dynamic career roadmaps, and manages an anonymous hierarchical grievance escalation system.
* Who will use it? Students, Teachers, HODs (Head of Departments), and Directors.
* What problem does it solve in real life? It bridges the gap between generic study materials and personalized tutoring. It eliminates manual test creation for teachers, provides immediate simplified AI-driven explanations strictly constrained to real syllabus materials, and ensures unresolved student complaints are securely escalated to higher authorities.
* Why did you choose this project? To leverage modern LLM capabilities (Google Gemini) within a realistic, highly authenticated, multi-tiered institutional framework bridging education and administrative oversight.

### 🔷 3. System Features (IMPORTANT)

👉 Your features:

* Adaptive Quiz System:
  How it works: Teachers input weak topics. The Gemini AI generates a 5-question MCQ test specifically focusing on that topic. Students take the test one time (QuizAttempt lockout), and the backend comprehensively evaluates their answers against the AI's answer key, automatically awarding leader-board credits.

* AI Chatbot (RAG):
  How it works: A Dual-Agent RAG (Retrieval-Augmented Generation) pipeline. The student clicks UI dropdowns to isolate a specific uploaded PDF. The backend extracts the document text. An 'Extraction Agent' scans the raw text for the rigid answer facts. Then, those facts are passed to the 'Simplification Agent' to translate the answer into plain, highly readable concepts for the student.

* Performance Analysis:
  How it works: The system records student scores in the `QuizAttempt` table and updates absolute scores in the `User` table under generic `credits`. These are aggregated into a globally ranked UI Leaderboard for gamified learning.

* Teacher Dashboard:
  How it works: Teachers access a secure portal to assign dynamic tests to their explicitly assigned classes, securely upload proprietary PDF study materials linked strictly to their subjects, and monitor active class sessions.

* Skill Development Module:
  How it works: The 'Learn New' tab generates dynamic career Roadmaps. When clicked, it builds an interactive UI where students learn sequentially via Section Gating. It dynamically scripts URLs connecting students to highly specific YouTube Tutorials and GeeksforGeeks query results based exactly on the AI's topics.

* Complaint System:
  How it works: A token-based anonymity system. Students submit grievances which generate an untraceable tracking token rather than linking their User ID. HODs review them. If an HOD ignores a complaint for strictly >15 days, the backend automatically mutates the complaint owner to "Director" and fires it up the executive sequence.

### 🔷 4. Backend Implementation (Flask)

* Did you use Flask? Yes
* Structure of project: Uses Modular Blueprints (`auth.py`, `student.py`, `teacher.py`, `hod_director.py`), a unified `models.py` for Database ORM, `templates/` for Jinja HTML mapping, and `ai_engine/` for isolated API pipeline logic.
* Major APIs you created:
  * `/login`: Handled via auth, validates roles and launches specific hierarchical session dashboards.
  * `/register`: Generic user creation and database relational committal.
  * `/student/chat`: API endpoint serving the async chatbot. Intercepts JSON payloads, accesses `PyMuPDF` for filesystem parsing, queries Gemini, and sends parsed outputs safely.
  * `/student/start_test & /submit_test`: API cluster rendering active subject tests dynamically, grading NLP submissions with whitespace-cleaning regex arrays.
* How authentication works? Uses `Flask-Login` session arrays. Dashboard/Route access is strictly segregated using `@login_required` hooks cross-referenced against strings explicitly matching `current_user.role`.

### 🔷 5. Database Design (VERY IMPORTANT 🚨)

* Which database? SQLite (Tracked and deployed via SQLAlchemy ORM).
* Tables you created:
  * users: Stores auth strings, institutional roles (Student/Teacher/etc.), and leaderboard `credits`.
  * test_session & questionbank: (quizzes) Maps generation of structural AI strings to active institutional blocks.
  * quiz_attempt: (results) Logs lockouts and unrepeatable success metrics for each active student.
  * complaints: The secure token table tracking grievance texts and their status metrics.
  * others: `Department`, `Class`, `Subject`, `Assignment` (Teacher-Subject network nodes), `StudyMaterial` (PDF filesystem mappings), `Roadmap` & `StudentProgress`.
* Relationships between tables? High relational density. E.g., A Teacher `User` relates to `Assignment`, which branches targeting both a `Subject` and a `Class`. `StudyMaterial` files are strictly back-referenced inside `Subject` scopes.
* Any ER diagram idea? A centralized hierarchical tree structure. The master `User` node branches outwards into mapping tables representing systemic academic constraints (Classes/Subjects) downwards into specific action logs (Attempts/Complaints).

### 🔷 6. Machine Learning Implementation

* Which algorithms you used? Advanced Generative AI (Large Language Models). Core implementation relies on `Google Gemini 1.5 Flash`. No classical ML trees (Random Forest/SVM) are deployed.
* What problem ML is solving in your project? Educational Content Generation (creating infinite sets of highly relevant quizzes dynamically) and Reading Comprehension (Parsing 100-page density PDF files to instantly answer hyper-specific contextual questions without hallucinations).
* Input data: Rigid API prompts injected with either raw manual Teacher keywords (topics) or raw extraction binary strings pulled out of PDF file caches.
* Output: Highly formatted JSON structures translated into Python via regex (Quiz Bank logic) or simplified instructional text arrays for live web chat.
* How model is trained? Utilizes Google's globally pre-trained Foundation mechanism. Customized behaviors are forced strictly through dynamic Prompt Engineering strategies (System Rules injection).
* Any libraries used? `google-generativeai`.

### 🔷 7. NLP / Chatbot (RAG)

* How chatbot works? It intercepts user questions, isolates the linked user document UI choice, reads the context, and restricts the LLM boundary logic to form a holistic conversational answer explicitly bounded by institutional data.
* Did you use:
  * embeddings? No.
  * vector database? No.
  * simple keyword search? No, utilizes Context Window Pumping.
* How documents are processed? The `PyMuPDF` library accesses the specific document, extracts string data recursively, and artificially bounds the array text block safely within 12,000 architectural characters.
* How response is generated? Via a proprietary Dual-Agent sequential loop. Agent 1 is invoked to silently strip unwanted document noise. The parsed array is then shipped explicitly to Agent 2 across the same API to generate the user-facing HTML format.

### 🔷 8. Multi-Agent System (IMPORTANT if used)

* What agents you created?
  * The Quiz generator: Located structurally in `routes/teacher.py`, enforces constraints and MCQ mapping.
  * The Extraction Agent: Agent 1 in the Chatbot pipeline deployed for precise fact mining against contextual queries.

* How they interact? Sequentially. The Extraction Agent's raw generic output string seamlessly passes in code architecture as the exact `Extracted Raw Content:` isolated input parameters mapping over into the Simplification Agent queue.

### 🔷 10. System Architecture (FLOW)

Step-by-step Execution Architecture:
1. User interacts with UI / Browser → 
2. DOM triggers Client-side Python Jinja Route inside Flask → 
3. Flask backend actively searches matching relational logic from SQLite / SQLAlchemy → 
4. Conditional logic determines if direct output occurs, OR triggers async API loop inside the LLM (`google-generativeai`) passing internal texts (`PyMuPDF`) → 
5. Backend parses arrays via Regex mappings → 
6. Processed clean dictionary is returned and injected back to dynamically update the UI state.

### 🔷 13. Challenges Faced

* What problems you faced? 
  - Cross-document API Hallucinations where generalized Subject RAG questions caused the AI to summarize arbitrary material structurally disconnected from the current class syllabus context limits.
  - Core system 500 error crashes when complex Gemini AI loops triggered 'Blank/Refused' content protocols over specific queries.
  - OS-agnostic slash interpretation faults parsing uploaded PDF location strings.
* How you solved them? 
  - Restructured RAG to require explicit front-end UI `material_id` mapping. 
  - Nested all Gemini triggers actively inside strict try/except python fail-safe blocks forcing static 'NOT_FOUND' default text behavior over server crashes.
  - Forced `.replace('\\', '/')` standardizing formatting structures.
